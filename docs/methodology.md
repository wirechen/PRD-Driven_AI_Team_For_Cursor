# PRD 驱动的 AI 团队开发方法论

## 一、为什么需要 PRD 驱动？

### 1.1 AI Agent 的记忆问题

AI Agent（如 Cursor 中的子代理）在每次会话中拥有有限的上下文窗口。跨会话时，上下文会丢失。这意味着：

- Agent 不记得上次做了什么
- Agent 不知道项目的全貌
- 口头传达的需求会在会话中遗失

**PRD 解决了这个问题**：它是一个持久化的、结构化的需求文档，任何 Agent 在任何时候都可以阅读它来了解项目的完整需求。

### 1.2 人机协作的最佳分工

| 角色 | 擅长 | 不擅长 |
|------|------|--------|
| **用户（你）** | 知道要什么、判断好不好 | 不想写代码、不想盯细节 |
| **AI Agent** | 快速编码、并行执行、不知疲倦 | 不知道你真正想要什么 |

PRD 就是你和 AI 之间的契约：你写清楚要什么，AI 严格按照执行。

### 1.3 核心原则

> **代码服从 PRD，PRD 服从用户。**

- 实现与 PRD 不一致 → Bug（开发修）
- PRD 与用户意图不一致 → 需求问题（产品经理修 PRD）
- 用户改了需求 → 先改 PRD，再改代码

## 二、多 Agent 团队架构

### 2.1 为什么需要多 Agent？

单个 Agent 同时做产品设计、前端开发、后端开发、测试，会导致：

- 上下文窗口被撑满，效率急剧下降
- 角色混乱，容易遗漏细节
- 无法并行，效率低下

分角色的多 Agent 团队解决了这些问题：

- 每个 Agent 专注一个领域，上下文更聚焦
- 不同 Agent 可以并行工作（配置 `is_background: true`）
- 职责清晰，产出物可追溯

### 2.2 主 Agent（项目经理）的关键作用

主 Agent 不直接编码，而是充当**通信中枢和调度器**：

```
用户 ←→ 主Agent ←→ 子Agent 1
                ←→ 子Agent 2
                ←→ 子Agent 3
```

这个设计基于 Cursor 平台的特性：
- 主 Agent 通过 `Task` 工具调用子 Agent
- 在同一条消息中发起多个 `Task` 可以实现真正并行
- 子 Agent 之间**不能直接通信**，必须通过主 Agent 中转

### 2.3 推荐的角色配置

| 角色 | 职责 | 为什么需要 |
|------|------|-----------|
| **产品经理** | 维护 PRD、设计功能 | 确保需求清晰、完整 |
| **前端工程师** | UI 设计 + 前端开发 | 专注页面和交互 |
| **后端工程师** | API + 数据库 | 专注业务逻辑和数据 |
| **测试工程师** | 接口测试 + E2E | 确保质量 |

## 三、PRD 的 7 个维度

经过实战验证，一个足够指导 AI 开发的 PRD 功能描述需要包含以下 7 个维度：

### 3.1 页面布局

描述页面各区域的结构和位置关系。

```markdown
**页面布局：**
- 顶部：4个统计卡片横向排列（待审核数/本月投票数/总户数/已认证数）
- 中部：最近的认证申请列表（最新5条）
- 底部：最近的投票列表（最新5条）
```

### 3.2 数据展示

列表/表格的具体列定义、字段格式。

```markdown
**数据展示：**
- 表格列：姓名 / 手机号 / 楼栋房号（格式：1栋2单元501）/ 申请时间 / 状态
- 状态标签颜色：已认证=绿色、待审核=橙色、已驳回=红色
```

### 3.3 用户交互

按钮行为、弹窗内容、表单校验。

```markdown
**用户交互：**
- 点击"删除"弹出确认弹窗，内容"确定删除？此操作不可恢复"
- 提交表单后按钮变为loading状态，禁止重复提交
```

### 3.4 状态流转

业务实体的状态变化。

```markdown
**状态流转：**
- 投票状态：draft → scheduled → active → ended → archived
- scheduled 状态：到达 start_time 后自动转为 active
```

### 3.5 异常处理

边界场景的提示文案。

```markdown
**异常处理：**
- 手机号重复：提示"该手机号已在本小区认证过，如有疑问请联系物业"
- 删除有投票记录的业主：提示"该业主已参与投票，无法删除"
```

### 3.6 接口依赖

列出前端需要调用的后端 API。

```markdown
**接口依赖：**
- `GET /api/v1/owners?status=pending` — 获取待审核列表
- `PUT /api/v1/owners/{id}/verify` — 通过认证
```

### 3.7 验收标准

可测试的验收条件。

```markdown
**验收标准：**
- [ ] 统计卡片数据与数据库一致
- [ ] 分页功能正常，每页20条
- [ ] 导出的Excel文件可正常打开
```

## 四、开发流程最佳实践

### 4.1 阶段化推进

不要试图一次性开发所有功能。将项目分为多个 Phase：

- **Phase 1**：基础架构（项目骨架、数据库、认证登录）
- **Phase 2**：核心功能模块
- **Phase 3**：完善和优化
- **Phase 4**：商业化/上线

每个 Phase 完成后必须经过完整的 **测试→Bug修复→回归→验收** 闭环。

### 4.2 并行工作模式

利用 `is_background: true` 配置，在同一条消息中启动多个子 Agent：

```
用户："开始 Phase 2 的楼栋管理和业主管理模块开发"

主 Agent 同时启动：
├── frontend-agent → 前端页面开发
├── backend-agent  → 后端接口开发
└── （完成后）test-agent → 测试
```

### 4.3 Bug 分流

测试工程师输出 Bug 列表后，主 Agent（项目经理）按规则分流：

- API 500/逻辑错误 → 后端
- 页面显示/交互问题 → 前端
- 前后端字段不一致 → 双方
- PRD 本身的问题 → 产品经理

## 五、实战踩坑指南

以下经验来自真实项目的开发过程，固化在子代理的工作规范中：

### 5.1 前后端字段名不一致

**问题**：前端用 `per_floor`，后端用 `rooms_per_floor`，导致功能不生效。

**规范**：前端 API 调用的方法（GET/POST/PUT/DELETE）和参数名**必须**与后端一致。后端接口定义优先。

### 5.2 HTTP 方法不一致

**问题**：后端用 `PUT` 实现 publish/verify 等操作，前端用了 `POST`，导致 405。

**规范**：操作类接口（发布、审核、结束等）统一用 `PUT`，创建用 `POST`。

### 5.3 ORM 枚举值大小写

**问题**：SQLAlchemy Enum 默认用大写 name（`ACTIVE`），但 MySQL 存储的是小写 value（`active`），查询报 `LookupError`。

**规范**：Enum 字段必须使用 `values_callable=lambda e: [m.value for m in e]`。

### 5.4 时区处理

**问题**：后端用 `DateTime(timezone=True)` 存储，SQLAlchemy 自动转 UTC，但 MySQL 的 DATETIME 是 naive 类型，导致时间差 8 小时。

**规范**：统一使用 naive datetime 存储（北京时间），比较时注意时区转换。

### 5.5 异步 ORM 的 MissingGreenlet

**问题**：在 async SQLAlchemy 中，`flush()` 后直接访问 server_default 字段报 `MissingGreenlet`。

**规范**：`await db.flush()` 之后必须 `await db.refresh(obj)` 才能安全读取字段。

### 5.6 子代理不能嵌套调用

**问题**：想让 test-agent 内部启动 browser-use 子代理做页面测试，但 Cursor 不支持子代理嵌套。

**规范**：test-agent 直接使用 Playwright 命令行做 E2E 测试，不通过子代理。所有子代理间的协作必须通过主 Agent 中转。

### 5.7 `is_background` 配置

**问题**：子代理默认同步执行（主 Agent 等待完成才能继续），无法并行。

**规范**：所有子代理必须在 frontmatter 中配置 `is_background: true`，这样主 Agent 可以在同一条消息中同时启动多个子代理。

### 5.8 WXML 模板语法限制

**问题**：小程序 WXML 不支持 JavaScript 数组索引语法 `{{name[0]}}`。

**规范**：需要在 JS 中预计算值，再传给模板渲染。

## 六、总结

PRD 驱动的 AI 团队开发模式，核心是三个字：**写清楚**。

你把需求写清楚（PRD），AI 团队就能：
- 产品经理把 PRD 写得足够详细
- 前端工程师严格按 PRD 实现页面
- 后端工程师严格按 PRD 实现接口
- 测试工程师按 PRD 验收标准测试
- 项目经理协调一切，并行推进

**你只需要关心 PRD，其余交给 AI 团队。**
